심심해서 써보는 뮤텍스

글을 보기 이전에

이 내용은 C언어의 멀티스레딩에 대하여 어느 정도의 사전지식이 필요합니다.

서론

C언어에서 어떤 작업을 동시에 처리할 때, 우리는 보통 멀티스레딩 헤더(pthread)를 불러와서 처리하게 된다. 이때, 각각의 스레드는 프로세스 안의 메모리를 공유하기 때문에 스레드 안에서도 전역 변수에 접근할 수 있다. 보통 이렇게 하면 프로그래밍은 훨씬 편해진다. 전역변수를 따로 스레드를 실행할 때 인자로 집어넣어야 하지 않아도 되니까. 그런데, 이렇게 프로그래밍 하면 스레드에서 전역 변수 값을 변경할 때 문제가 발생할 수가 있다. 다음 코드를 보자:

xxxxxxxxxx
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>
​
#define ADDTIMES 1000000
​
int a = 0;
​
void *addToA() {
    for(int i = 0; i < ADDTIMES; i++) {
        a += 1;
    }
}
​
int main() {
    pthread_t threads[2];
    int error;
    for (int i = 0; i < 2; i++) {
        error = pthread_create(&threads[i], NULL, addToA, NULL);
        if (error) {
            printf("Error: code %d\n", error);
            exit(-1);
        }
    }
    
    for(int i = 0; i < 2; i++) {
        pthread_join(threads[i], NULL);
    }
    printf("value of a: %d", a);
    return 0;
}

이 코드는 전역 변수 a에 두 스레드가 1씩 1000000번 더하는 소스로, 언뜻 보면 출력 결과가 2000000이 되어야 할 것 같다. 하지만 직접 실행을 해 보면:

xxxxxxxxxx
value of a: 1316598
Process returned 0 (0x0)   execution time : 0.124 s
Press any key to continue.

이런 결과가 나온다. 기대한 값과는 상당히 다른 것 같다.

왜 이럴까?

앞선 프로그램에서는 결과값이 기대한 2000000이 아닌, 그 이하의 값이 나왔다. 왜 이럴까?

11번째 줄의

xxxxxxxxxx
a += 1;

을 보자.

이 곳을 Disassembly 해보면 다음과 같은 코드가 나온다.

  • 12번째 줄에서는 현재 a의 주소(여기에서는 0x412008)에서 a를 가져온다.
  • 13번째 줄에서는 가져온 값에 1을 더한다.
  • 14번째 줄에서는 현재 a의 주소에 더한 값을 넣는다.

보기에는 완벽한 것 같지만, 여기에는 맹점이 있다. 스레드 두 개가 동시에 실행되기 때문에, 위의 코드도 동시에 두 개가 실행된다는 점이다(물론 완전히 같은 코드는 아니다. 하지만 적어도 12~14번째 줄은 두 스레드에서 같다). 이 때, 충돌이 일어난다.

이상적인 실행 순서는 다음과 같다:

  1. 1번 스레드가 a의 주소에서 값을 가져온다.
  2. 1번 스레드에서 1을 더한다.
  3. 1번 스레드에서 값을 a의 주소에 넣는다.
  4. 2번 스레드가 a의 주소에서 값을 가져온다.
  5. 2번 스레드에서 1을 더한다.
  6. 2번 스레드에서 값을 a의 주소에 넣는다.

(1~3번과 4~6번의 순서가 바뀔 수는 있다)

하지만, 우리의 프로그램에서는 두 스레드가 동시에 실행되기 때문에 12~14번째 줄의 코드가 다음과 같이 섞일 수 도 있다:

  1. 1번 스레드가 a의 주소에서 값을 가져온다.
  2. 2번 스레드가 a의 주소에서 값을 가져온다.
  3. 1번 스레드에서 1을 더한다.
  4. 2번 스레드에서 1을 더한다.
  5. 1번 스레드에서 값을 a의 주소에 넣는다.
  6. 2번 스레드에서 값을 a의 주소에 넣는다.

1번이 실행되기 전의 a 값을 1000이라고 하자.

  • 1번 스레드는 a에서 1000을 가져온다. 마찬가지로 2번 스레드도 a에서 1000을 가져온다.
  • 1번 스레드에서 1을 더하여 값이 1001이 되었다. 2번 스레드에서도 1을 더하여 값이 1001이 되었다.
  • 1번 스레드에서 값(1001)을 a의 주소에 넣는다.
  • 2번 스레드에서 값(1001)을 a의 주소에 넣는다.

이 떄 a의 값은 두 번 더했을 때의 결과 1002가 아닌, 1001이 되어 버린다. 분명 더한 건 두 번이지만, 값은 한 번만 증가해 버린 것이다.

해결 방안, 뮤텍스

뮤텍스, mutex은 상호 배제(mutual exclusion)의 약자이다. 쉽게 말해서, "내가 이 영역을 쓸 테니 다른 스레드는 사용하지 마시오" 라고 스레드가 말하는 것이다. 화장실 자물쇠를 생각해보면 쉽다. 화장실에 들어가서 볼 일을 볼 때, 문을 잠그면 밖에서는 잠금 표시를 보고 들어오려고 하지 않고 안의 사람이 나올 때까지 기다린다. 이때 화장실 자물쇠를 뮤텍스라고 할 수 있을 것이다. 그러면 뮤텍스를 적용한 코드를 보도록 하자:

xxxxxxxxxx
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>
​
#define ADDTIMES 1000000
​
int a = 0;
pthread_mutex_t toiletLock; // 자물쇠는 하나만 있어도 충분하다
​
void *addToA() {
    for(int i = 0; i < ADDTIMES; i++) {
        pthread_mutex_lock(&toiletLock); // 잠군다
        a += 1;
        pthread_mutex_unlock(&toiletLock); // 연다
    }
}
​
int main()
{
    pthread_t threads[2];
    int error;
​
    if (pthread_mutex_init(&toiletLock, NULL)) {
        printf("Lock has some problem");
    }
​
    for (int i = 0; i < 2; i++) {
        error = pthread_create(&threads[i], NULL, addToA, NULL);
        if (error) {
            printf("Error: code %d\n", error);
            exit(-1);
        }
    }
    for(int i = 0; i < 2; i++) {
        pthread_join(threads[i], NULL);
    }
    pthread_mutex_destroy(&toiletLock); // 더 이상 자물쇠를 쓸 일이 없으니 부수자
    printf("value of a: %d", a);
    return 0;
}

1을 더하는 13번째 줄 근처를 Disassembly 해보자:

0x401370    add    $0x1,%eax
0x401373    mov    %eax,0x40d008
;14 :           pthread_mutex_unlock(&toiletLock);

; --후략--

4번째 줄 pthread_mutex_lock에서는 뮤텍스가 다른 스레드에 의하여 잠겨 있는지, 잠겨 있지 않은지 확인한다. 잠겨 있다면, 들어갈 수 없으므로 열릴 때 까지 기다리고 열리면 자신이 뮤텍스를 잠근다.

11번째 줄에 와서 1을 더하는 작업이 끝나면, 잠긴 뮤텍스를 풀고 다른 스레드가 뮤텍스를 잠글 수 있도록 한다.

이 프로그램의 실행 결과는 다음과 같다:

xxxxxxxxxx
value of a: 2000000
Process returned 0 (0x0)   execution time : 36.597 s
Press any key to continue.

보이는 대로, 값은 2000000이며 프로그램은 원하는 대로 작동했다.

뮤텍스는 만능인가?

원하는 대로 결과가 나오니, 멀티 스레딩 프로그래밍을 할 때 뮤텍스를 자주 쓰자라고 생각할 수 있다. 하지만, 사실 뮤텍스를 자주 쓰는 건 좋지 않다. 위 프로그램의 실행 결과를 보면, 실행시간이 무려 40초에 육박하는 매우 무거운 프로그램이라는 것을 알 수 있다. 대부분의 시간을 서로 자물쇠를 풀고 잠그는 데에 사용한 것이다. 그러면 멀티 스레딩을 하면서 프로그램을 의도 대로 작동시키려면 어떻게 해야 할까?

  • 알고리즘을 최적화한다.

    보이다 시피 위 코드는, 아무런 의미도 없다. 굳이 멀티 스레드로 쓸 필요도 없다. 자신의 알고리즘이 필요 이상으로 무거운 것은 아닌지 한번 생각해 보자.

  • 멀티 스레딩 대신 멀티 프로세싱을 한다.

    일반적으로, 멀티 프로세싱은 멀티 스레딩보다는 느리다. 하지만, 뮤텍스 연산을 불가피하게 많이 작동 시켜야 하는 경우가 있을 수도 있다. 이때는 멀티 프로세싱이 더 빠를 수도 있다.

결론

사실 C언어는 상당한 저수준 언어이다. 멀티 스레딩을 하여 병렬로 데이터를 처리하려고 해도, 고려해야 할 것이 상당히 많다. 지금 사용한 언어만 봐도, 우리는 C언어의 작동을 이해하기 위하여 기계어인 어셈블리까지 손을 대어야 했다. 물론 잘 짜면 다른 거의 모든 언어보다도 빠르게 짤 수 있지만, 사용자가 미숙하다면 프로그램 자체가 작동하지 않을 수도 있다(당장 어셈블리어 보려고 디버거 돌릴 때 DRM 프로그램때문에 디버거가 동작하지도 않았다). 다른 프로그래밍 언어를 이용하자. 멀티스레딩 프로그래밍에서는 항상 여러 상황을 고려하고, 예외를 생각할 수 있어야 한다. 특히 C언어 같은 저수준 언어에서는 더.
